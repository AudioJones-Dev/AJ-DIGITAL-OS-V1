# AJ Digital OS - Approval System Specification
**Version:** 1.0  
**Updated:** 2026-04-24  
**Owner:** AJ DIGITAL LLC  
**Status:** Implementation-ready scaffold

---

## 1. Purpose

The Approval System is the human authorization layer for high-risk and privileged actions generated by agents.

It ensures gated actions do not execute until an explicit approval decision exists.

---

## 2. Architecture

Runtime path:

Permission Enforcement -> Approval Service -> Approval Store -> Decision -> Execute or Block

Components:

- `approval-types.ts`: typed schema for approvals
- `approval-store.ts`: in-memory store abstraction (replaceable by SQLite/Postgres later)
- `approval-service.ts`: create, approve, deny, expire, cancel, fetch, and validate approvals
- `enforcement-engine.ts`: creates approval requests when decisions are approval-gated

## 4. Schema

```json
{
  "approvalId": "string",
  "requestedAt": "ISO timestamp",
  "expiresAt": "ISO timestamp",
  "requestedByAgentId": "string",
  "permissionLevel": 4,
  "actionCategory": "GIT_PUSH",
  "risk": "high",
  "reason": "Remote push modifies shared repository state.",
  "target": "origin/main",
  "command": "git push origin main",
  "clientId": null,
  "environment": "dev",
  "status": "pending",
  "approvedBy": null,
  "approvalChannel": null,
  "auditId": "uuid-or-null"
}
```

---

## 5. Runtime Flow

1. Enforcement classifies and policy-checks action.
2. If action requires approval and no valid approval is supplied, create approval request.
3. Return `require_approval` with reason and audit/approval linkage.
4. Operator approves or denies through channel.
5. Approved request can be reused until expiry.

---

## 6. Examples

- Git push at level 4 without approval -> pending approval request.
- Secret modify at level 5 without approval -> pending approval request.
- Deployment approval request without environment -> rejected by service input validation.

---

## 7. Tests

Current tests cover:

- creates approval request
- approves pending request
- denies pending request
- expired approval cannot be used
- production deployment approval requires explicit environment field

---

## 8. Limitations

- in-memory store only
- no cross-process persistence
- no approval UI yet
- no channel delivery implementation (telegram/dashboard/email only modeled as enums)

---

## 9. Future Implementation Path

1. Replace in-memory store with SQLite/Postgres adapter.
2. Add approval notification dispatchers by channel.
3. Add approval signatures and replay protection.
4. Add policy-based expiry durations by risk.

---

## 10. Integration

With Agent Permission Enforcement:

- enforcement engine auto-creates pending approvals on `require_approval` decisions.

With Security / Trust Layer:

- approval requirements become enforceable runtime controls, not documentation-only guidance.
